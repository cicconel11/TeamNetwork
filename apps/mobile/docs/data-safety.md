## 2. Camera as non-required feature

### Status: âś… Done in this branch, verify in next build

### What changed

Added `apps/mobile/plugins/withOptionalHardwareFeatures.js` â€” a
`withAndroidManifest` config plugin that injects the following into the
merged Android manifest on every `expo prebuild`:

```xml
<uses-feature android:name="android.hardware.camera" android:required="false" />
<uses-feature android:name="android.hardware.camera.any" android:required="false" />
<uses-feature android:name="android.hardware.camera.autofocus" android:required="false" />
<uses-feature android:name="android.hardware.camera.front" android:required="false" />
```

Registered in `apps/mobile/app.json â†’ expo.plugins` as
`"./plugins/withOptionalHardwareFeatures"`.

### Why

Declaring the `CAMERA` permission implicitly marks `android.hardware.camera`
as **required**, which makes Play filter out devices without a camera
(some Chromebooks, some tablets, and most emulators). Explicitly setting
`required="false"` widens device compatibility without changing runtime
behavior â€” the app still asks for camera permission when it actually needs
the camera; it just doesn't exclude camera-less devices at install time.

### Verify on the next build

1. Run `npx expo config --type introspect` locally and confirm the four
   `uses-feature` entries are present with `android:required="false"`.
2. After the next EAS build, upload the `.aab` to Play Console and check
   **Internal testing â†’ bundle details â†’ Features**. We should see the
   camera features listed, and the **"Supported devices"** count should go
   up vs the current build (which reports `camera` as required).
3. If the count does not change, re-check the plugin output and make sure
   `expo prebuild` is not regenerating the manifest after the plugin runs.

### Why a plugin, not a manifest edit

`apps/mobile/android/` is gitignored and regenerated on every EAS build, so
any direct edits to `AndroidManifest.xml` would be wiped. A plugin runs
during `expo prebuild` on the EAS builder and writes the same change every
time.

## 3. Trim `USE_FINGERPRINT` if unused

### Status: âš ď¸Ź Open â€” needs code audit

### What the current build declares

From the `.aab` we just uploaded (internal testing, versionCode 3):

- `android.permission.USE_BIOMETRIC` â€” current biometric API (API 28+)
- `android.permission.USE_FINGERPRINT` â€” **deprecated** in API 28, replaced
  by `USE_BIOMETRIC`

Having both is harmless but messy, and `USE_FINGERPRINT` signals to
reviewers and users that we might be using deprecated APIs.

### Investigation needed

1. **Who pulls these in?** Neither permission is declared in `app.json`
   explicitly, so they come from a dependency. Likely suspects:
   - `expo-local-authentication` â€” if installed, it pulls `USE_BIOMETRIC`
     on newer versions and may also carry `USE_FINGERPRINT` for back-compat.
   - A security/auth library with fingerprint fallback.

   Run on the next prebuild:
   ```bash
   cd apps/mobile
   npx expo prebuild --clean --no-install --platform android
   grep -rn "USE_FINGERPRINT\|USE_BIOMETRIC" android/app/src/main/AndroidManifest.xml android/app/build/intermediates/merged_manifests 2>/dev/null
   ```
   Then check the merger report at
   `android/app/build/outputs/logs/manifest-merger-*-report.txt` â€” it
   tells you which library introduced which permission.

2. **Does the app actually use biometric auth?** Grep the source:
   ```bash
   rg -n "LocalAuthentication|expo-local-authentication|authenticateAsync" apps/mobile/src apps/mobile/app
   ```
   - If there are no matches: we don't use biometrics at all. Remove the
     dependency (if it's a direct dep) or block both permissions in
     `app.json â†’ android.blockedPermissions`.
   - If there are matches but only `USE_FINGERPRINT` is unused: block just
     that one.

3. **If we do use biometrics:** keep `USE_BIOMETRIC`, block
   `USE_FINGERPRINT` explicitly:
   ```json
   "blockedPermissions": [
     "...existing entries...",
     "android.permission.USE_FINGERPRINT"
   ]
   ```
   This removes it from the merged manifest via Android manifest-merger
   `tools:node="remove"`.

### Acceptance

After the next build, the `.aab` bundle details in Play Console should
list **either** `USE_BIOMETRIC` alone **or** neither permission.
`USE_FINGERPRINT` should be gone.

## 4. Biometric + camera audit

### Status: âš ď¸Ź Open â€” tied to Â§3 investigation

Goal: make sure the permissions we declare match what the app actually
uses at runtime. Declaring permissions we don't use is a Play Store red
flag and makes the install dialog look sketchier than it needs to.

### Camera

1. **Where is `CAMERA` requested in code?**
   ```bash
   rg -n "ImagePicker\.(requestCamera|launchCamera)|MediaLibrary\.|Camera\." apps/mobile/src apps/mobile/app
   ```
2. **Expected usages:**
   - Profile photo capture (`expo-image-picker â†’ launchCameraAsync`)
   - Announcement image attachment
   - Any QR scanner screen (is there one? `react-native-qrcode-svg` is
     render-only, so probably not â€” confirm no camera-based QR reader).
3. **Decision:**
   - If the app has zero `launchCameraAsync` call sites, we don't need
     `CAMERA` at all and can block it. The `expo-image-picker` photo
     library API does not require `CAMERA`.
   - If we do use it, keep `CAMERA` and make sure every screen that opens
     the camera also shows a rationale string before the system prompt.

### Biometrics

Covered by Â§3. The code audit step (`rg LocalAuthentication`) is the same
for both questions â€” do it once and resolve Â§3 and Â§4 together.

### Notifications (adjacent concern worth auditing)

While we're checking: confirm that `POST_NOTIFICATIONS` is requested at the
right moment. Current behavior should be:

1. First app open â†’ no prompt.
2. User enters a screen where they'd clearly benefit from notifications
   (e.g. an org they just joined, or Settings â†’ Notifications).
3. Call `Notifications.requestPermissionsAsync()` there, not at launch.

Prompting at launch is the single most common reason users deny push, and
Play reviewers have been known to flag it as "unclear purpose."
