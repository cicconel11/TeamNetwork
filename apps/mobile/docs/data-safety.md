## 2. Camera as non-required feature

### Status: ✅ Done in this branch, verify in next build

### What changed

Added `apps/mobile/plugins/withOptionalHardwareFeatures.js` — a
`withAndroidManifest` config plugin that injects the following into the
merged Android manifest on every `expo prebuild`:

```xml
<uses-feature android:name="android.hardware.camera" android:required="false" />
<uses-feature android:name="android.hardware.camera.any" android:required="false" />
<uses-feature android:name="android.hardware.camera.autofocus" android:required="false" />
<uses-feature android:name="android.hardware.camera.front" android:required="false" />
```

Registered in `apps/mobile/app.json → expo.plugins` as
`"./plugins/withOptionalHardwareFeatures"`.

### Why

Declaring the `CAMERA` permission implicitly marks `android.hardware.camera`
as **required**, which makes Play filter out devices without a camera
(some Chromebooks, some tablets, and most emulators). Explicitly setting
`required="false"` widens device compatibility without changing runtime
behavior — the app still asks for camera permission when it actually needs
the camera; it just doesn't exclude camera-less devices at install time.

### Verify on the next build

1. Run `npx expo config --type introspect` locally and confirm the four
   `uses-feature` entries are present with `android:required="false"`.
2. After the next EAS build, upload the `.aab` to Play Console and check
   **Internal testing → bundle details → Features**. We should see the
   camera features listed, and the **"Supported devices"** count should go
   up vs the current build (which reports `camera` as required).
3. If the count does not change, re-check the plugin output and make sure
   `expo prebuild` is not regenerating the manifest after the plugin runs.

### Why a plugin, not a manifest edit

`apps/mobile/android/` is gitignored and regenerated on every EAS build, so
any direct edits to `AndroidManifest.xml` would be wiped. A plugin runs
during `expo prebuild` on the EAS builder and writes the same change every
time.

## 3. Trim `USE_FINGERPRINT` if unused

### Status: ✅ Blocked in `app.config.ts` (2026-05-19) — verify on next Android build

### Decision

The app DOES use biometric auth via `expo-local-authentication`
(`apps/mobile/src/lib/biometric.ts:15-72` calls `hasHardwareAsync`,
`isEnrolledAsync`, `authenticateAsync`; consumed by
`apps/mobile/src/components/settings/SettingsSecuritySection.tsx:6-9,85`
and the biometric-lock overlay at
`apps/mobile/src/contexts/BiometricLockContext.tsx:31-34`).

Modern Android requires only `USE_BIOMETRIC` (API 28+). `USE_FINGERPRINT`
is the deprecated pre-API-28 fallback and is not needed. Both
historically appeared in the merged manifest via the
`expo-local-authentication` dependency. We now strip the deprecated one.

### What changed

`apps/mobile/app.config.ts:92-98` — `USE_FINGERPRINT` added to
`android.blockedPermissions`:

```ts
blockedPermissions: [
  "android.permission.RECORD_AUDIO",
  "android.permission.READ_EXTERNAL_STORAGE",
  "android.permission.WRITE_EXTERNAL_STORAGE",
  "android.permission.SYSTEM_ALERT_WINDOW",
  "android.permission.USE_FINGERPRINT",
],
```

This injects `tools:node="remove"` for that permission via Android
manifest-merger.

### Verify on the next Android EAS build

iOS submission (build 29) does not exercise this. On the next Android
build:

1. Run `expo prebuild --clean --no-install --platform android` and grep
   `android/app/src/main/AndroidManifest.xml` for `USE_FINGERPRINT` —
   expect 0 hits (or a `tools:node="remove"` line).
2. After uploading the `.aab` to Play Console, **Internal testing →
   bundle details → Permissions** should list `USE_BIOMETRIC` only.

## 4. Biometric + camera audit

### Status: ✅ Resolved 2026-05-19 (camera + biometric). ⚠️ Notifications-on-launch audit still open.

### Camera — IS used at runtime

CAMERA permission is genuine, not transitive. Used exclusively for QR
scanning (org join + event check-in), not photo capture:

- `apps/mobile/src/components/QRScanner.tsx:3,28` — imports `CameraView`
  from `expo-camera`, renders the scanner UI.
- `apps/mobile/src/lib/device-permissions.ts:107-108,150-152` —
  `Camera.getCameraPermissionsAsync()` /
  `Camera.requestCameraPermissionsAsync()`.
- `apps/mobile/app/(app)/(drawer)/[orgSlug]/events/[eventId]/scan.tsx:7,288` —
  renders `<QRScanner>` for event check-in.

No `launchCameraAsync` call sites exist (no photo capture). Matches the
`NSCameraUsageDescription` string at `app.config.ts:56-57`:

> "Scan a TeamNetwork QR code to join your organization or check members
> in at events."

`CAMERA` permission stays. Profile/post photos come from the
`expo-image-picker` photo library API which does not require `CAMERA`.

### Biometrics — IS used at runtime

Resolved in §3. `expo-local-authentication` powers biometric unlock
(`apps/mobile/src/lib/biometric.ts`, `BiometricLockContext`,
`SettingsSecuritySection`). Merged manifest now declares `USE_BIOMETRIC`
only — `USE_FINGERPRINT` blocked.

### Notifications — ⚠️ Open

Confirm `POST_NOTIFICATIONS` is requested at the right moment.
Expected:

1. First app open → no prompt.
2. User enters a screen where notifications matter (org join, settings).
3. Call `Notifications.requestPermissionsAsync()` there, not at launch.

Prompting at launch is the most common reason users deny push, and Play
reviewers have flagged it as "unclear purpose." iOS reviewers usually
accept either pattern but in-context prompting has higher accept rates.

Check: `apps/mobile/src/hooks/usePushNotifications.ts`. Not blocking
iOS build 29 submission.
