Design decisions

decisionU1 — guards live as plain functions, not a hook

Spec asked for a mobile-local mirror of web's consumeClaimRate + email-confirmed check. I exposed two plain exports — consumeClaimRate(userId, now?) and assertEmailConfirmed(user) — plus an internal __resetClaimRateForTests() escape hatch. Functions (no hook, no context) match the web shape exactly and let the Jest suite reset state between tests without faking timers. The now? parameter is a test-only seam; production callers omit it.

decisionU1 — guards execute after an explicit getUser() call

Web's claimAlumniProfile() calls supabase.auth.getUser() first, then checks email_confirmed_at, then rate-limit, then RPC. Mobile follows the same order: verifyOtp → getUser → assertEmailConfirmed → consumeClaimRate → rpc("claim_alumni_profiles"). getUser after verifyOtp is one extra round-trip the spec did not strictly require, but it is the only way to read email_confirmed_at without trusting the in-memory session (which may be stale if the user was already signed in). It also mirrors web behavior precisely.

decisionU1 — RPC error surface masked to "Failed to claim alumni profile"

Web masks RPC errors to that exact string and logs userId + message. Mobile now does the same: captureException includes { screen: "Claim", step: "rpc", userId }, the UI shows "Failed to claim alumni profile". Previously mobile showed "Could not finish claiming. Please try again." — that copy is gone. If you want to keep the friendlier copy, change one string in claim.tsx.

decisionU2 — exempt only on the second-segment claim, not by URL string

Existing pattern uses segments[1] === "callback". Reproduced literally: segments[1] === "claim". This couples the exemption to the file-system route name app/(auth)/claim.tsx; if that file is renamed, the exemption breaks silently. Acceptable — the existing callback/reset-password checks share the same fragility.

decisionU3 — only the claim path is lowercased, not the whole pathname

I compute const claimPath = parsed.pathname.toLowerCase() and match on that variable only. PKCE callback path stays case-sensitive, code parameters stay untouched. Matches the spec verbatim.

decisionU4 — redirect wins over the 1-org "land on org home" branch

The spec said "when redirectParam starts with /, prefer it over /(app)/(drawer)/${orgs[0].slug}." I lifted the redirect branch above the 1-org branch in navigatePostClaim, so the priority order is: 0 orgs → fallback join screen (with redirect threaded into query), redirect present → use redirect, 1 org → org home, ≥2 orgs → drawer root.

decisionU5 — .replace(/\D/g, "").slice(0, 8) already does the right thing

The existing onChangeText already strips non-digits and slices to 8. Removing maxLength={8} is the entire fix; no other change to the handler.

decisionU6 — only the two /auth/claim entries switch to pathPattern

Spec said "All other filters unchanged." Implemented as such; auth/callback remains pathPrefix (Supabase callback paths are well-bounded by the ?code= query, no neighboring paths to confuse).

decisionU7 — native claim case in parseNativeRoute just delegates

Replaced the inline 7-line redirect-sanitization with a one-line return parseClaimPayload(url);. parseClaimPayload uses only url.searchParams.get(...), which is identical between https: and teammeet: URL objects, so this is a true no-op refactor for behavior.

decisionU8 — DeviceID purpose = Analytics; Email + Name = AppFunctionality

Spec specified these three identifiers + purposes. Email and Name are entered by the user and drive sign-in / membership lookup → AppFunctionality. DeviceID (Sentry + PostHog identifiers) → Analytics. Linked = true on all three because we associate them with the authenticated user id. Tracking flag stays false.

decisionU9 — timeout fires once per handleRequestSubmit

Spec placed the timer in a useEffect that "calls turnstileRef.current?.show()." There is no such effect in the existing screen — the captcha is opened imperatively from handleRequestSubmit. I put the timer there instead: every Send Code tap clears any previous timer and starts a fresh 15s one. On success (handleCaptchaVerify), cancel (handleCaptchaCancel), error (handleCaptchaError), or unmount, the timer is cleared. Matches R9 intent without forcing a render-on-mount effect that would fire even before the user has tapped Send.

decisionU10 — signup payload matches login shape

Login uses { screen: "Login", method: "email" }; spec asked for { screen: "Signup", method: "email" } on signup. Implemented verbatim. Error string Turnstile: ${message} matches claim + login.

decisionU11 — schema lives in @teammeet/validation, validated at the mobile call site

Schema: z.array(z.object({ out_organization_id: z.string().uuid(), out_slug: z.string().min(1) })). Exported as claimedOrgRowSchema + ClaimedOrgRow type. Used via safeParse(rpcData ?? []) in claim.tsx; failure is logged + masked to the same generic copy as RPC errors. Web does not consume this schema yet; it can adopt later without code churn here.

Deviations from the spec

deviationU8 PrivacyInfo edit is ephemeral — not committed

Spec assumed apps/mobile/ios/TeamNetwork/PrivacyInfo.xcprivacy is a tracked file. It is not: apps/mobile/.gitignore ignores all of /ios/ as a prebuild artifact (git ls-files apps/mobile/ios/ returns nothing). I made the EmailAddress + Name + DeviceID edits to the file on disk so a current local archive would include them, but those edits will be blown away on the next bun run mobile:prebuild:clean. The existing 5 entries must come from a template or post-prebuild manual edit that nobody tracked down yet.

Correct fix (deferred): add a small Expo config plugin (e.g. plugins/withPrivacyManifestExtras.js) that mutates PrivacyInfo.xcprivacy in onPostPrebuild to inject the three additional entries on every prebuild. That requires a separate code review and a follow-up PR; I would rather flag the gap than ship something that silently regresses on next clean prebuild.

Confirm: file the follow-up issue, or have me write the config plugin now.

deviationNo validation-package test file

The spec asks for packages/validation/src/__tests__/claim.test.ts. The validation package has no test runner configured (its package.json exposes only typecheck; no Jest, no Vitest, no test deps). Adding a runner is out of scope for this PR (it would touch tooling, CI, and add devDependencies). The schema is exercised indirectly by mobile typecheck and by the mobile claim-screen integration path. Confirm: accept the skip + file a follow-up, or move schema tests into apps/mobile/__tests__/lib/ instead.

deviationTest path for U1 is flat, not nested

Spec named the test at apps/mobile/__tests__/lib/auth/claim-guards.test.ts. The mobile __tests__/lib/ directory uses a flat convention (no subdirs); I matched it and put the file at apps/mobile/__tests__/lib/claim-guards.test.ts. No functional impact.

deviationU9 timeout is not in a useEffect

See the U9 decision above — the screen has no captcha-mount effect to hang the timer on. Net behavior matches R9: stalled captcha surfaces an error within ≤15s; cleanup on unmount via the existing single useEffect(() => () => clearCaptchaTimeout(), []).

deviationRPC error mask copy changed

Previous mobile copy: "Could not finish claiming. Please try again.". Spec said to mirror web's "Failed to claim alumni profile". I picked the spec wording. If product prefers the friendlier sentence, this is a one-string edit.

Tradeoffs considered

tradeoffRate-limit storage: in-memory Map vs AsyncStorage

Picked in-memory Map. Resets on process restart, identical to web's per-process Map. AsyncStorage would survive app kills but adds I/O on the hot path and a separate concern (TTL bookkeeping, race conditions on concurrent reads). Intent is defense-in-depth, not anti-abuse; persistence is not worth the complexity. Server-side enforcement in the RPC is the real fix and is deferred per the spec.

tradeoffGuards inlined vs single runClaimRPC wrapper

Considered wrapping guards + RPC + parse into one helper to slim down handleVerifySubmit. Rejected — the call site is ~25 lines, each early-return surfaces a different toast string, and inlining keeps the error mapping visible without indirection. CLAUDE.md "Simplicity First": no abstraction for single-use code.

tradeoffU4 redirect priority — above or below the 1-org branch?

Two valid orderings:

  1. Redirect wins universally (picked). Honors the user's explicit intent even with one org.
  2. 1-org branch wins. Treats "land on your org" as the primary destination, uses redirect only for ambiguous cases.

R4's text — "Honor redirect on every post-claim navigation branch (0 orgs, 1 org, ≥2 orgs)" — argues for #1. If the deep link came from an event email (?redirect=/acme-hs/events/evt-1), bouncing to the org home would lose the event context. Picked #1.

tradeoffU9 timeout cleanup: per-handler vs centralized state machine

Added clearCaptchaTimeout() at the top of every captcha lifecycle handler (Verify, Cancel, Error) plus a useEffect cleanup. Alternative was wiring it into a single state machine. Picked per-handler — costs four extra lines and the read order matches the spec's verification scenarios one-for-one.

Open questions

questionShould the validation-package test file land in this PR?

Currently skipped because the package has no test runner. Options: (a) accept skip + follow-up issue, (b) add Vitest to the package now (devDep + CI wiring), (c) put schema tests in apps/mobile/__tests__/lib/. I recommend (a). Happy to do (c) on a one-line nudge.

questionRPC failure copy — "Failed to claim alumni profile" vs prior friendlier copy?

Used the spec wording. The web string reads like a developer-facing server error; on a phone screen "We couldn't finish claiming. Please try again." reads more naturally. Confirm preference.

questionU9 timeout window — 15s sufficient?

Spec set 15s. On a poor cellular connection Turnstile can take longer. Current handling: timeout treated as a soft error, captcha widget dismissed, user can retry. If product prefers a longer window (e.g. 25s) or a softer "still loading" hint at 15s + hard fail at 30s, one-constant edit.

questionU2 deep-link UX while signed in

A signed-in user tapping a claim link now lands on the claim screen with a session already attached. If they enter a different email, signInWithOtp + verifyOtp will silently swap the session. Documented web behavior, but on mobile the resulting session swap may surprise the user (signed in as X, now signed in as Y). Should we show a "You are signed in as X — claiming a different email will replace your session" confirmation? Web does not. Defaulting to "no, mirror web."
